# bigbag-advisor

Радник підбору біг-бегів + маршрутизатор клієнтських запитів у відділи. Демо-конфіг
налаштований під реальну компанію — Торгову Групу **«ВБА»** ([vba.com.ua](https://vba.com.ua/uk/)),
виробника біг-бегів, вапняку та мінерального порошку.

**Що це робить:** менеджер (або клієнт на сайті) пише вільним текстом —
*«беги під соняшник, тонна, возитимемо вагонами»* — система повертає:

1. **типову специфікацію біг-бега** (розмір, стропи, вкладиш, верх, Q-бег),
2. **у який відділ** передати запит (біг-беги / вапняк-порошок / загальний),
3. **чернетку відповіді** клієнту.

> Усі рекомендації ґрунтуються на **реальному контенті сайту ВБА** (сторінки продукції).
> Це не вигадані поради — це формалізація того, що ВБА вже пише клієнтам вручну.

---

## Навіщо це, чесно

Це не «ще один чат-бот». Суть проєкту — **виміряти, де AI реально потрібен, а де
вистачає простих правил**. Підбір біг-бега виглядає як задача для LLM, але насправді
сторінка ВБА — це майже детермінована таблиця: тип вантажу → специфікація. Тому
правильна інженерна позиція — спершу побудувати тупий baseline на правилах, а LLM
додавати, **тільки якщо він виміряно б'є його**.

Для бізнесу тут є конкретна ціна помилки: вигадана специфікація (розмір, якого ВБА не
шиє) = неправильне замовлення = втрачені гроші. Тому ми окремо міряємо **галюцинації**.

## Виміряні результати

Бенч відтворюваний: фіксований датасет (`src/eval/dataset.ts`, 33 кейси UA/RU/EN +
пастки), один скрипт `npm run bench`.

### Rules-baseline (без LLM, без API, безкоштовно)

| Метрика | Результат |
|---|---|
| Маршрутизація у відділ | **100% (33/33)** |
| Класифікація типу вантажу | **100% (22/22)** |
| Стропи (де однозначно) | **100% (6/6)** |
| Вкладиш (де однозначно) | **100% (7/7)** |
| Галюцинації (spec поза каталогом) ↓ | **0% (0/33)** |
| Вигадав spec на не-біг-бег запит ↓ | **0% (0/11)** |
| Вартість / латентність | **$0 / миттєво** |

**Чесне застереження про ці 100%.** Цифри отримані на тому ж наборі, проти якого я
**вручну доганяв словник синонімів**. Коли я вперше прогнав бенч, було 88% роутингу і
82% категорії — падало на RU-формах («бэги», «нитроаммофоску», «гранитный отсев») і на
конфлікті «соняшник» (зерно) vs «соняшниковий жмих» (макуха). Я додав ці синоніми — і
вийшло 100%. **Це не означає, що система ідеальна.** Це означає рівно одне: правила
дешеві, але **крихкі до лексики, якої не передбачив автор**. На held-out наборі з новою
лексикою діри з'являться знову. Саме ця крихкість — головна ціна правил, і саме
питання «чи прибирає її LLM, не починаючи галюцинувати» вимірює наступний крок.

### LLM (grounded vs no-ground) — ⏳ ще не виміряно

Код готовий (`src/core/llm.ts`, режими `llm` і `llm-no-ground`), але **я не публікую
цифр, яких не виміряв**. Щоб отримати реальну таблицю порівняння:

```bash
cp .env.example .env          # впишіть OPENROUTER_API_KEY
npm run bench                 # прожене rules + llm + llm-no-ground
```

Гіпотеза (буде підтверджена або спростована числами, не обіцянками):
- **grounded LLM** зрівняється з правилами на точності, але коштуватиме гроші й мс — тобто
  для цієї структурованої задачі він **надлишковий**;
- **no-ground LLM** покаже помітний % галюцинацій (вигадані розміри типу `85x85`), бо без
  каталогу ВБА він тягне «загальні» знання про FIBC;
- реальна користь LLM — на **брудному вільному тексті й нових мовах**, де правила
  потребують ручного словника.

Якщо числа спростують гіпотезу — у README поїде правда, а не зручна версія.

## Запуск

```bash
npm install
npm run dev            # веб: http://localhost:3000
```

- `/` — внутрішній **sales-desk** для менеджера/новачка.
- `/widget` — спрощений **клієнтський віджет** для сайту.

Бенч:

```bash
npm run bench:rules    # тільки правила (без ключа)
npm run bench          # + LLM (потрібен OPENROUTER_API_KEY)
```

## Архітектура

```
data/
  knowledge-base.json     # джерело правди: правила підбору, каталог, відділи (з сайту ВБА)
  raw/                    # сирий текст сторінок ВБА (провенанс — звідки взято факти)
src/core/
  rules.ts                # RULES-baseline: ключові слова → категорія, відділ, специфікація
  llm.ts                  # LLM через OpenRouter: grounded / no-ground
  kb.ts                   # завантаження KB + детектор «поза каталогом» (= галюцинація)
  types.ts
src/eval/
  dataset.ts              # 33 розмічені кейси (UA/RU/EN + пастки)
  bench.ts                # rules vs llm vs llm-no-ground → таблиця метрик
src/app/                  # Next.js: /desk, /widget, /api/advise (одне ядро на обидва)
```

Ядро (`src/core`) не залежить від Next — його однаково імпортують і веб, і CLI-бенч.

**Детектор галюцинацій** (`kb.ts → outOfCatalog`) детермінований: будь-яка специфікація
звіряється з допустимим каталогом ВБА (розміри дна, кількість петель, вантажопідйомність
500–2000 кг, щільність 110–200 г/м²). Значення поза множиною = галюцинація. Тому ми
ловимо вигадки навіть у LLM — без «LLM-судді».

## Як це лягає на роботу всередині компанії

Покривається кілька реальних процесів ВБА одним ядром:

- **економія часу менеджера** — типову специфікацію не треба тримати в голові;
- **навчання новачка** — бот = жива база знань відділу продажів;
- **маршрутизація** — запит одразу йде у правильний відділ (на сайті ВБА відділи й так
  розділені: біг-беги / вапняк+порошок);
- **AI-помічник для клієнтів** — той самий рушій як віджет на сайті.

Розгортання: внутрішній веб для менеджерів, або вбудований `<iframe>` віджет на сайт.
LLM-шар вмикається лише там, де правила виміряно програють — не «бо AI».

## Обмеження (чесно)

- Датасет малий (33) і курований автором; це демонстрація методу, не продакшн-валідація.
- Специфікація — **типова**, не остаточна: ВБА шиє під замовлення, фінальні розміри
  узгоджує менеджер. Радник це проговорює в кожній відповіді.
- Rules-словник синонімів треба доповнювати під нову лексику (саме тут потенційно
  допоможе LLM — буде видно з бенчу).
- Ціни моделей у `llm.ts` — орієнтовні, для оцінки порядку вартості.
