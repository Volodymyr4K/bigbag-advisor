🇺🇦 Українська · [🇬🇧 English](README.en.md)

Радник підбору біг-бегів + маршрутизатор клієнтських запитів у відділи. Демо-конфіг
налаштований під реальну компанію — Торгову Групу **«ВБА»** ([vba.com.ua](https://vba.com.ua/uk/)),
виробника біг-бегів, вапняку та мінерального порошку.

**Що це робить:** менеджер (або клієнт на сайті) пише вільним текстом —
*«беги під соняшник, тонна, возитимемо вагонами»* — система повертає:

1. **типову специфікацію біг-бега** (розмір, стропи, вкладиш, верх, Q-бег),
2. **у який відділ** передати запит (біг-беги / вапняк-порошок / загальний),
3. **чернетку відповіді** клієнту.

> Усі рекомендації ґрунтуються на **реальному контенті сайту ВБА** (сторінки продукції).
> Це не вигадані поради — це формалізація того, що ВБА вже пише клієнтам вручну.

## Навіщо це, чесно

Це не «ще один чат-бот». Суть проєкту — **виміряти, де AI реально потрібен, а де
вистачає простішого**. Підбір біг-бега виглядає як задача для LLM, але насправді
сторінка ВБА — це майже детермінована таблиця: тип вантажу → специфікація. Тому
правильна інженерна позиція — пройти **драбину знизу вгору**: спершу тупі правила, потім
класичний ML, і лише потім LLM — і на кожному щаблі **міряти, чи вищий щабель того вартий**.
Антихайп тут не гасло, а виміряний результат: на наших даних дорожчий ML правила не б'є
(див. цифри нижче).

Для бізнесу тут є конкретна ціна помилки: вигадана специфікація (розмір, якого ВБА не
шиє) = неправильне замовлення = втрачені гроші. Тому ми окремо міряємо **галюцинації**.

## Виміряні результати

Драбина рішень — **rules → класичний ML → LLM** — на двох наборах. Бенч відтворюваний
одним скриптом (`npm run bench`):

- **DEV** (`src/eval/dataset.ts`, 33 кейси) — набір, на якому я тюнив словник правил.
  Тут у правил «домашня перевага»: вони бачили цю лексику.
- **HELD-OUT** (`src/eval/heldout.ts`, 44 свіжих кейси) — **головний** набір. Цих рядків
  правила не бачили, ML на них не тренувався. Є навмисно невидана лексика («ріпак»,
  «суперфосфат», «лушпиння», «гравій», «керамзит», «нут»). Саме тут видно реальну генералізацію.

ML — чистий TypeScript, multinomial Naive Bayes на символьних n-грамах (`src/ml/`),
навчений на **синтетичному** наборі (`src/ml/traindata.ts`).

### DEV (домашня перевага правил)

| Engine | Відділ | Категорія | Галюцинації ↓ |
|---|---|---|---|
| rules | **100% (33/33)** | **100% (22/22)** | 0% |
| ml (NaiveBayes) | 85% (28/33) | 86% (19/22) | 0% |

### HELD-OUT (свіжа лексика — чесна картина)

| Engine | Відділ | Категорія | Галюцинації ↓ | $/100 | Латентн. |
|---|---|---|---|---|---|
| rules | **98% (43/44)** | 48% (14/29) | 0% | $0 | 0ms |
| ml (NaiveBayes) | 84% (37/44) | **59% (17/29)** | 0% | $0 | 0ms |
| llm (grounded) | ⏳ | ⏳ | ⏳ | ⏳ | ⏳ |
| llm (no-ground) | ⏳ | ⏳ | ⏳ | ⏳ | ⏳ |

👉 **По-кейсовий розбір** кожного held-out промаху (хто що передбачив, ✓/✗):
[`docs/heldout-report.md`](docs/heldout-report.md) — генерується `npm run report`.
Видно нюанс: на «ріпак» ML вгадав `grain`, а на «суперфосфат» помилково дав `oilcake`.

### Що ці числа чесно кажуть

1. **100% правил на DEV — це була ілюзія.** На HELD-OUT класифікація категорії падає до
   **48%**. Правила тримаються рівно на тій лексиці, яку автор передбачив; нові слова
   («ріпак», «суперфосфат», «керамзит») їх ламають. Це виміряна ціна підходу «ключові слова».
2. **ML дає скромну перевагу — але не порятунок.** Категорія: **59% (17/29) проти 48%
   (14/29)** у правил. На n=29 це вже не чистий шум: char-n-грами інколи генералізують на
   невидане (на «ріпак» ML вгадав `grain`). АЛЕ у роутингу ML **гірший** (84% проти 95%),
   і обидва провалюють більшість нової лексики. Чесний висновок: ML на синтетичних даних
   **не дає однозначного виграшу** — десь краще, десь гірше. Його суперсила (вчитися з
   даних) нема на чому проявитись: реального розміченого корпусу ще нема.
3. **Маршрутизація тримається краще за класифікацію** (84–98%): відрізнити «біг-бег vs
   вапняк» простіше, ніж точний тип вантажу.
4. **Галюцинацій 0% скрізь** — бо і rules, і ML беруть специфікацію лише з каталогу VBA,
   фізично не можуть вигадати розмір. Це закладено в архітектуру, не в везіння.

**Звідси справжнє питання** (а не «ML чи LLM хайп»): коли в тебе **нема даних**, вибір —
між дешевими крихкими правилами і zero-shot LLM, що тягне нову лексику, але коштує грошей
і може галюцинувати. LLM-рядки вище — ⏳ бо **я не публікую цифр, яких не виміряв**.

```bash
cp .env.example .env          # впишіть OPENROUTER_API_KEY
npm run bench                 # додасть рядки llm / llm-no-ground у таблицю
```

Очікування (буде підтверджено або спростовано числами): grounded LLM підніме категорію
на HELD-OUT помітно вище 48–59% (zero-shot тягне нові слова), а no-ground LLM покаже
галюцинації (вигадані розміри поза каталогом). Якщо ні — у README поїде правда.

## Запуск

```bash
npm install
npm run dev            # веб: http://localhost:3000
```

- `/` — внутрішній **sales-desk** для менеджера/новачка.
- `/widget` — спрощений **клієнтський віджет** для сайту.

### Вбудова віджета на сайт ВБА

`?embed=1` прибирає зайвий «хром» (заголовок/футер) — для чистого `<iframe>`:

```html
<iframe
  src="https://YOUR-HOST/widget?embed=1"
  title="Підбір біг-бега"
  style="width:100%;max-width:760px;height:640px;border:0"
  loading="lazy">
</iframe>
```

Бенч:

```bash
npm test               # тести ключової логіки (без API)
npm run bench:offline  # rules + ML (без ключа, безкоштовно)
npm run bench          # + LLM (потрібен OPENROUTER_API_KEY)
npm run report         # по-кейсовий розбір held-out → docs/heldout-report.md
npm run scrape         # перезібрати сторінки ВБА + перевірити «маяки» (провенанс)
```

## Архітектура

```
data/
  knowledge-base.json     # джерело правди: правила підбору, каталог, відділи (з сайту ВБА)
  raw/                    # сирий текст сторінок ВБА (провенанс — звідки взято факти)
src/core/
  rules.ts                # RULES-baseline: ключові слова → категорія, відділ, специфікація
  ml.ts                   # ML-рушій: класифікатор → категорія, далі spec з KB
  llm.ts                  # LLM через OpenRouter: grounded / no-ground
  kb.ts                   # завантаження KB + детектор «поза каталогом» (= галюцинація)
  misslog.ts              # flywheel: лог low-confidence запитів (майбутній корпус для ML)
  types.ts
scripts/
  scrape.ts               # провенанс: перезбір сторінок ВБА + перевірка «маяків»
src/ml/
  classifier.ts           # multinomial Naive Bayes на char n-грамах (чистий TS)
  traindata.ts            # СИНТЕТИЧНІ тренувальні дані (чесно позначені)
src/eval/
  dataset.ts              # DEV: 33 кейси (UA/RU/EN + пастки) — тут тюнились правила
  heldout.ts              # HELD-OUT: 44 свіжих кейси з невиданою лексикою
  bench.ts                # rules vs ml vs llm vs llm-no-ground → дві таблиці метрик
  report.ts               # по-кейсовий розбір held-out → docs/heldout-report.md
src/app/                  # Next.js: /desk, /widget, /api/advise, /api/misses
```

Ядро (`src/core`) не залежить від Next — його однаково імпортують і веб, і CLI-бенч.

**Детектор галюцинацій** (`kb.ts → outOfCatalog`) детермінований: будь-яка специфікація
звіряється з допустимим каталогом ВБА (розміри дна, кількість петель, вантажопідйомність
500–2000 кг, щільність 110–200 г/м²). Значення поза множиною = галюцинація. Тому ми
ловимо вигадки навіть у LLM — без «LLM-судді».

## Як це лягає на роботу всередині компанії

Покривається кілька реальних процесів ВБА одним ядром:

- **економія часу менеджера** — типову специфікацію не треба тримати в голові;
- **навчання новачка** — бот = жива база знань відділу продажів;
- **маршрутизація** — запит одразу йде у правильний відділ (на сайті ВБА відділи й так
  розділені: біг-беги / вапняк+порошок);
- **AI-помічник для клієнтів** — той самий рушій як віджет на сайті.

Розгортання: внутрішній веб для менеджерів, або вбудований `<iframe>` віджет на сайт.
LLM-шар вмикається лише там, де правила виміряно програють — не «бо AI».

**Flywheel збору даних.** Виміри показали головну дірку: класичний ML голодний, бо
**розміченого корпусу звернень ще нема**. Тому система сама його збирає: кожен запит, де
вона не дала впевненої спец (низька впевненість / відділ «загальний»), тихо логується
([`src/core/misslog.ts`](src/core/misslog.ts) → `data/miss-log.jsonl`, переглянути:
`GET /api/misses`). Через місяці це стає реальним датасетом, на якому ML нарешті матиме що
вчити. Чесно про два обмеження: на serverless (Vercel) файлова ФС ефемерна — у проді лог
треба на БД; а ендпоінт віддає тексти запитів клієнтів, тож перед публічним деплоєм його
закриває env `MISSES_TOKEN` (без нього відкритий — лише для локалі).

📄 Детальний план під ВБА (процеси, інструкції для персоналу, як міряти ефективність,
де AI **не** треба): [`docs/vba-rollout.md`](docs/vba-rollout.md).

## Обмеження (чесно)

- Датасети малі (DEV 33, HELD-OUT 44) і куровані автором; навіть на n=29 категорійних
  кейсів числа орієнтовні. Це демонстрація **методу**, не продакшн-валідація.
- ML навчений на **синтетичних** даних (реального корпусу звернень у бізнесу ще нема) —
  тому його «програш» правилам очікуваний і чесний, а не вирок класичному ML загалом.
- Специфікація — **типова**, не остаточна: ВБА шиє під замовлення, фінальні розміри
  узгоджує менеджер. Радник це проговорює в кожній відповіді.
- Rules-словник синонімів треба доповнювати під нову лексику (саме тут потенційно
  допоможе LLM — буде видно з бенчу).
- Ціни моделей у `llm.ts` — орієнтовні, для оцінки порядку вартості.

## Частина серії (portfolio thesis)

Цей проєкт — частина серії з єдиною темою: **чесно міряти, де AI реально допомагає, а де
вистачає простішого (правила / класичний ML), і не ліпити LLM заради хайпу.** Спільний
підхід: фіксований датасет, baseline знизу, метрики (точність, галюцинації, вартість),
відтворюваний бенч.

- [**clauselens**](https://github.com/Volodymyr4K/clauselens) — аналіз юридичних пунктів з евалом.
- [**deskbench**](https://github.com/Volodymyr4K/deskbench) — front-desk помічник для сервісного бізнесу, eval-first.
- [**bigbag-advisor**](https://github.com/Volodymyr4K/bigbag-advisor) — *цей*: повна драбина rules → ML → LLM на held-out.
- [market-efficiency-lab](https://github.com/Volodymyr4K/market-efficiency-lab) · [ownfleet](https://github.com/Volodymyr4K/ownfleet) · [vendora](https://github.com/Volodymyr4K/vendora) — суміжні проєкти.

Чим **bigbag-advisor відрізняється** від двох сусідів: це єдиний, де пройдена *повна*
драбина рішень (правила → класичний ML → LLM) і де виміряно на **held-out** наборі, що
дорожчий щабель не завжди вартий того.

## Стек

Next.js (App Router) · TypeScript · класичний ML чистим TS (без Python/залежностей) ·
LLM через OpenAI-сумісний endpoint (OpenRouter).
Без БД: база знань — версіонований JSON, щоб правки міг робити не-програміст.
